Purpose of This DocumentThis document defines **how RAE and its overlays (including RAE-CRL)** are deployed and executedacross different environments, while preserving:- architectural agnosticity,- privacy guarantees,- reproducibility,- low operational friction.RAE must support **both Docker-based and non-Docker deployments** without changing core logic.---## 2. Core Deployment Principle (Non-Negotiable)&gt; **Deployment mode must never affect RAE semantics.**This means:- same memory model,- same artifact schema,- same Mesh policy behavior,- same privacy guarantees.Only **operational concerns** may differ.---## 3. Supported Execution ModesRAE must support the following execution modes:1. **Native (No Docker)**2. **Dockerized**3. **Hybrid (optional components in Docker)**All modes must be first-class citizens.---## 4. Native (No Docker) Mode### 4.1 Target Use Cases- Arch Linux (research labs)- RAE Mobile- Air-gapped environments- Windows standalone builds- Low-overhead systems### 4.2 Characteristics- direct binary / Python execution- no container runtime required- full offline support- minimal system dependencies### 4.3 Requirements- RAE core must:  - auto-detect filesystem paths,  - auto-detect hardware (RAM / CPU / GPU),  - gracefully degrade (Lite → Full optional).- Configuration:  - single local config file  - environment-variable override allowed### 4.4 LLM Handling (Native)- optional- local-only- executed as:  - separate process  - strict input/output contract- zero network access---## 5. Docker Mode### 5.1 Target Use Cases- CI / CD- Team deployments- Reproducible experiments- Server-side RAE-Full- Controlled Mesh gateways### 5.2 Characteristics- fully containerized execution- reproducible environments- explicit resource limits- network isolation by default### 5.3 Docker Design Rules- RAE core runs in its own container- LLM (if present) runs in a **separate container**- Mesh communication isolated via explicit network rules- No container may auto-scale memory or network access### 5.4 Required Docker Artifacts- Dockerfile:  - minimal base image  - no unnecessary tooling- docker-compose.yml:  - RAE core  - optional LLM  - optional Mesh gateway- volumes:  - explicit memory persistence  - no implicit data sharing---## 6. Hybrid Mode (Optional but Recommended)Hybrid mode allows:- RAE core running natively- heavy components (LLM, embeddings, indexing) in DockerTypical use case:- researcher laptop (native)- heavy model offloaded to local containerRules:- RAE core must function if containers are stopped- failure of container must not break RAE core---## 7. Storage and Database AgnosticityRAE and RAE-CRL must be agnostic to:- storage engine (SQLite, Postgres, others)- vector backend- filesystem layoutDeployment mode MUST NOT hardcode:- database paths- mount locations- container-specific assumptionsAll paths must be resolved at runtime.---## 8. Mesh Considerations Across Deployment Modes- Mesh protocol is deployment-neutral- Native and Docker nodes can interoperate- Privacy and sharing policy enforcement is identicalDocker mode MAY:- act as Mesh relay- enforce stricter auditingNative mode MUST:- require explicit user confirmation for all Mesh actions---## 9. RAE-CRL Specific GuaranteesRAE-CRL must:- run identically in Docker and non-Docker modes- not assume container presence- not assume shared filesystem- not assume persistent network accessAll CRL artifacts are stored via RAE core only.---## 10. Platform Mapping (Canonical)| Platform | Default Mode | Notes ||--------|--------------|------|| RAE-Mobile | Native | Privacy absolute || RAE-Windows | Native | Optional Hybrid || RAE-CRL (Research) | Native | Arch-friendly || Team RAE | Docker | Reproducibility || CI / Testing | Docker | Deterministic |---## 11. Explicit Non-GoalsThis deployment model does NOT:- force containerization,- assume cloud availability,- optimize for orchestration systems,- require Kubernetes.RAE remains **local-first and user-controlled**.---## 12. Validation CriteriaDeployment implementation is correct if:- the same CRL artifact behaves identically in all modes,- switching Docker ↔ Native does not require data migration,- Mesh policies remain enforced,- RAE-Lite always works as a fallback.---## 13. Summary StatementRAE supports multiple deployment modes **without semantic divergence**.Containers are an operational convenience —  **never a conceptual dependency**.
